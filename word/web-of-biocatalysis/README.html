Web of Biocatalysis

A living backbone of biocatalysis: continuously updated, interpreted, and structured into 10 pillars.

This repository contains the full pipeline, caches, and renderers that turn literature into living, citation-anchored pillar pages. It also documents how “foundation” JSON packs are fed to an external API (e.g., GPT‑5 online) using per‑section prompts for deterministic, auditable writing and polishing.

Directory layout

  • backbone/pillars/ — Pillar content and templates
    • 02_design_engineering/
      • template.md — page template with {{section:...}} placeholders
      • definition.md — stable definition seed text (optional)
      • sections/
        • sections.yaml — order and enable/disable per section
        • generated/ — compiled section markdown (deterministic or LLM-free stubs)
        • prompts.yaml — optional LLM prompts (local Ollama flow)
  • literature/ — data, caches, tools
    • tools/ — scripts, e.g. generate_pillars.py, build_pillar_foundation.py
    • index/ — global CSVs (papers.csv, index.csv)
    • cache/ — TEI-derived caches, pillar caches, core review caches
    • foundations/ — foundation JSON packs per pillar (e.g., 02_foundation.json)
    • prompts/ — canonical external API prompts (see below; newly formalized)

End‑to‑end workflow

  1. Ingest and index literature (PDF → TEI → index)
  1. Build or refresh pillar caches (TEI → pillar cache)

This happens on‑demand during generation, but you can refresh explicitly:

  1. Deterministic compile of pillar pages (no LLM)

Compiles sections from existing data and stubs, then stitches via the template.

  1. Foundation JSON for external API writing

Produce a foundation pack (evidence, bundles, seeds) without generating text:

  1. External API drafting/polishing (per‑section prompts)
  • Feed literature/foundations/02_foundation.json plus a section‑specific MASTER prompt to your external API (e.g., GPT‑5 online) to get a draft.
  • Optionally feed the draft through a Final Polish prompt (per section) to refine prose while preserving citations.
  • Save the final section text to backbone/pillars/02_design_engineering/sections/generated/<section>.md.
  1. Final stitch from foundation (LLM‑free compile)

This writes the final page to backbone/pillars/02_design_engineering/pillar-02-design_engineering.md, honoring sections/sections.yaml for order and enabled flags.

Section controls

  • File: backbone/pillars/02_design_engineering/sections/sections.yaml
  • Keys:
    • order: list of section names in render order (must match filenames in generated/, e.g., overview.md, key_papers.md).
    • enabled: boolean flags per section. Disabled sections are removed (header + placeholder) from the template during stitch.

Where to store external prompts (canonical)

  • New, standardized location for API prompts and polishers:
    • literature/prompts/pillar02/overview_master_prompt.md
    • literature/prompts/pillar02/overview_polish_prompt.md
  • Store similar files per pillar and per section (e.g., pillar07/history_master_prompt.md).
  • Rationale: prompts live alongside foundations/, versionable and independent of backbone/ content.

Stitching details and template

  • Template path: backbone/pillars/02_design_engineering/template.md
  • Placeholders: {{section:overview}}, {{section:key_papers}}, etc.
  • Generator: literature/tools/generate_pillars.py
    • Reads sections.yaml for order and enabled flags.
    • Loads section files from sections/generated/*.md.
    • Removes disabled sections (header + placeholder) and writes final page.

Caches and indices

  • Global index: literature/index/papers.csv (primary) and literature/index/index.csv (legacy)
  • Pillar cache: literature/cache/pillars/<ID>.json
  • Core reviews cache: literature/cache/core_reviews/core_reviews_v2.json

Notes and conventions

  • Section filenames must match keys in sections.yaml (overview.md, metrics.md, etc.).
  • Keep polished external outputs under sections/generated/ only; do not edit the compiled pillar file directly.
  • Store prompts in literature/prompts/<pillarXX>/<section>_*_prompt.md to keep them versioned and reusable.
